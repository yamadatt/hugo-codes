Goプロジェクトを新規に立ち上げる際、ディレクトリ構成やパッケージ分けに迷うことが多いでしょう。Goには「公式フレームワーク」が存在しないため、パッケージの命名や依存関係の設計ミスが循環参照（Circular Dependency）などの重大な設計破綻を招きます。今回はGoにおける健全なパッケージ構成の考え方を整理します。

<!--more-->

## 1. パッケージ分割のアンチパターン：レイヤーごとの分割

MVCやクリーンアーキテクチャの構成をそのままパッケージ名にしてしまう設計は、Goでは失敗しやすい代表例です。

```
# 良くない構成
mypkg/
  ├─ controllers/
  ├─ models/
  └─ views/
```

### なぜ良くないのか？
例えば `models` パッケージ内にすべてのドメインオブジェクトを置くと、異なるドメイン同士が互いを参照し合いやすく、コードの密結合と循環参照を招きます。また、パッケージ内に `user.go`、`product.go` など多数の無関係な要素が混在し、認知的負荷が高まります。

## 2. 健全な設計：ドメインや関心事による分割

Goでは、各パッケージが特定の「自己完結した機能」を持つように設計し、パッケージ自体が一種のAPIとして振る舞うようにするのがベストプラクティスです。

```
# 推奨される構成
myproject/
  ├─ cmd/
  │    └─ app/            # アプリケーションのエントリーポイント (main.go)
  ├─ internal/
  │    ├─ user/           # ユーザー関連のロジック、リポジトリ、API
  │    ├─ payment/        # 決済処理関連のロジック
  │    └─ db/             # 共通のデータベースコネクション管理
  └─ pkg/                 # 外部プロジェクトから再利用可能な汎用ユーティリティ
```

## 3. internal パッケージによるカプセル化

Goには `internal` という名前のディレクトリ配下のパッケージは、その親ディレクトリからしかインポートできないという、言語レベルのアクセス制御機能があります。
外部（パブリック）に公開する必要のない独自ドメインロジックは、すべて `internal` の下に置くことで、予期せぬモジュールの結合を防ぎ安全性を高めることができます。

## まとめ

Goのパッケージ設計の根底にあるのは「シンプルさ」と「明確な境界」です。レイヤーで細分化しすぎるのではなく、ビジネスドメインや単一責任原則に従ってパッケージを構築し、循環参照のない綺麗な依存関係を目指しましょう。
