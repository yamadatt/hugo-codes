Goで構造体を定義する際、フィールドを定義する「順番」によって構造体全体のメモリサイズが変化することをご存知でしょうか。これは、CPUがメモリに効率良くアクセスするための「アライメント（Alignment）」という仕組みによるものです。今回はアライメントとメモリサイズ最適化の方法を解説します。

<!--more-->

## 1. アライメントとパディングの仕組み

CPUは通常、メモリを1バイトずつではなく、4バイトや8バイト（32bit/64bit）といった「ワード」単位で読み込みます。これに最適化するため、コンパイラはメモリ上のデータを型に応じた境界に配置します。

例えば、`int8`（1バイト）の後ろに `int64`（8バイト）を配置すると、コンパイラは `int64` の配置位置を8の倍数のアドレスにするために、自動的に不要な空きメモリ（パディング）を挿入します。

```go
// 構造体のサイズが大きくなってしまう例
type BadStruct struct {
	A int8  // 1バイト
	// 7バイトのパディング（隙間）が自動挿入される
	B int64 // 8バイト
	C int8  // 1バイト
	// 7バイトのパディングが自動挿入される
}
```
この構造体の実質データは 1+8+1=10 バイトですが、パディングを含めた実際のメモリサイズは **24バイト** になります。

## 2. フィールド順序の変更による最適化

フィールドを「サイズの大きい順」または「同サイズをまとめる」ように並び替えるだけで、無駄なパディングを劇的に削減できます。

```go
// 最適化された配置
type GoodStruct struct {
	B int64 // 8バイト
	A int8  // 1バイト
	C int8  // 1バイト
	// 6バイトのパディングのみが最後に挿入される
}
```
この場合、構造体全体のメモリサイズは **16バイト** に縮小されます。フィールドの順序を変えるだけで、約33%のメモリ節約になります。

## 3. サイズの測定方法

実際にコードでサイズを検証するには、`unsafe.Sizeof` や `unsafe.Alignof` を使用します。

```go
package main

import (
	"fmt"
	"unsafe"
)

func main() {
	fmt.Println("BadStruct size:", unsafe.Sizeof(BadStruct{}))   // 24
	fmt.Println("GoodStruct size:", unsafe.Sizeof(GoodStruct{})) // 16
}
```

## まとめ

通常はそれほど意識する必要はありませんが、数百万個の構造体オブジェクトをスライスに保持してメモリに展開するようなケースでは、アライメントの最適化を行うだけで、メモリ消費量とキャッシュヒット率が改善し大きなパフォーマンス向上が得られます。
