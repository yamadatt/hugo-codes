並行プログラミングにおいて、最も発見が難しく深刻なバグの1つが「データレース（Data Race）」です。Goには、このデータレースをランタイム実行時に自動検知する強力な機能 `race detector` が組み込まれています。今回はその使い方を解説します。

<!--more-->

## 1. データレースとは？

データレースは、複数のゴルーチンが同期（ロックやチャネル等）を行わずに、同じメモリ領域に対して同時にアクセスし、かつ少なくとも1つが書き込み処理である場合に発生します。

```go
package main

import (
	"sync"
)

func main() {
	var count int
	var wg sync.WaitGroup

	for i := 0; i < 1000; i++ {
		wg.Add(1)
		go func() {
			defer wg.Done()
			count++ // 複数のゴルーチンから同時に書き込まれ、競合する
		}()
	}

	wg.Wait()
}
```

このコードを実行すると、`count++` の操作がアトミックではないため、最終的な値が `1000` にならないだけでなく、予期せぬメモリの破壊が発生することがあります。

## 2. `-race` フラグによる競合検出

テストやビルド時に `-race` フラグを付与するだけで、競合検出用のバイナリが生成されます。

### テスト実行時の検出
```sh
go test -race ./...
```

### アプリケーション実行時の検出
```sh
go build -race -o myapp
./myapp
```

このフラグをつけて実行すると、データレースが発生した瞬間に、スタックトレースと共に競合を知らせる警告情報がコンソールに出力され、どのコードが競合しているかをすぐに特定できます。

## 3. Race Detector のコストと注意点

Race detector は非常に便利ですが、実行時の検証コードが追加されるため、以下のコストがかかります。

- **メモリ消費量**: 通常時の約5〜10倍に増加します。
- **実行時間**: 通常時の約2〜20倍に低下します。

このため、**本番環境で常用するバイナリに `-race` を指定することは推奨されません**。ステージング環境やCIでのテストフェーズ、あるいはローカルテストでのみ使用するようにしましょう。

## まとめ

Goでゴルーチンを使うコードを書く場合、CIやローカルテストで `-race` フラグ付きのテストを実行することを習慣化しましょう。これにより、本番環境で発生する予測困難な並行処理バグを事前に防ぐことができます。
