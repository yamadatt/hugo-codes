# Go Notes 記事作成スキル

このリポジトリは日本語のGo技術ブログ「Go Notes」（Hugo製）。記事は必ず日本語で書く。

コンテンツは2種類ある。依頼内容からどちらかを判断する（迷ったら通常記事）。

| 種類 | 置き場所 | 用途 |
| --- | --- | --- |
| 通常記事 | `content/blog/<slug>.md` | 解説記事（60〜80行程度、コード例つき） |
| スニペット (bits) | `content/bits/<slug>.md` | 短いTips（数行の本文のみ） |

slugはケバブケースの英語で、既存記事にならい `go-` で始める（例: `go-channel-directions.md`）。

## 1. 事前確認（必須）

記事を書く前に以下を確認する。

```bash
# 既存記事の一覧を確認(テーマが重複しないslug・内容を選ぶ)
ls content/blog/

# 最新の記事日付と今日の日付を確認（日付は全記事で一意にする）
grep -rh "^date:" content/blog/*.md | sort | tail -3
date +%Y-%m-%d

# 使用済みアイコンを確認(アイコンは記事ごとに一意にする)
grep -rh "^icon:" content/blog/*.md | sort

# 利用可能なアイコン一覧（この中からのみ選ぶ。存在しない名前はビルド警告になる）
ls assets/icons/ui/

# 直近記事のhighlight色を確認（近い記事同士で同じ色が続かないようにする）
grep -rh "^highlight:" content/blog/*.md | sort | uniq -c
```

## 2. 通常記事のフロントマター

```yaml
---
title: Goの <span>Keyword</span> を使った〜〜
title_safe: GoのKeywordを使った〜〜
highlight: blue
pack: duotone
icon: code
draft: false
toc: true
comments: false
date: 2026-07-07
category: Go
tags:
  - go
  - keyword
---
```

各フィールドの規則:

- **title**: 記事の核となるキーワード（技術用語）を1つだけ `<span>...</span>` で囲む。spanの前後に半角スペースを入れる。
- **title_safe**: titleからspanタグを除いた素のテキスト（スペースも詰める）。
- **highlight**: Tailwindの色名。使用実績: `amber` `blue` `cyan` `emerald` `fuchsia` `green` `indigo` `lime` `orange` `pink` `purple` `red` `rose` `sky` `slate` `teal` `violet` `yellow`。直近の記事と重ならない色を選ぶ。
- **pack**: 常に `duotone`。
- **icon**: `assets/icons/ui/*.svg` に実在するファイル名（拡張子なし）。**既存記事で未使用のものを選ぶ**（1で確認済みのリストと突き合わせる）。
- **date**: **未来日付は禁止**（Hugoはデフォルトビルドで未来日付の記事を除外し、サイトに表示されない）。既存の最新日付が今日より前なら「最新日付の翌日〜今日」の空いている日を `YYYY-MM-DD` 形式で使う。最新日付がすでに今日なら、時刻付きの `YYYY-MM-DDTHH:00:00+09:00` 形式（例: `2026-07-06T12:00:00+09:00`）で現在時刻より前の時刻を付けて一意にする。
- **category**: `Go` 固定。
- **tags**: 小文字ケバブケース。先頭は必ず `go`、続けて記事テーマのタグを1〜2個。

## 3. 通常記事の本文構成

```markdown
（リード文: 記事の背景と「今回は〜を整理します」で締める1段落）

<!--more-->

## セクション1（基本の説明）

説明文と実行可能なGoコード例。

## セクション2〜4（応用・パターン・注意点）

## まとめ

（要点を2〜3文で）
```

- 見出しは `##`（h2）と `###`(h3)のみ使う(目次はh2〜h3を対象にしている)。
- リード文の直後に必ず `<!--more-->` を置く（一覧ページの抜粋区切り）。
- コードブロックは ```` ```go ```` で囲み、可能な限り `package main` から始まる実行可能な完結した例にする。コード内コメントは日本語。
- 分量はファイル全体（フロントマター含む）で**110行以内**。書き終えたら `wc -l` で確認し、超えていたら冗長なコード例や重複する説明を削る。コード例は3〜4個、セクションは「まとめ」を含めて4〜6個に抑える。
- 箇条書きで「使うべき場面／避けるべき場面」のような対比を入れると既存記事のトーンに合う。
- 誇張表現（「爆速」「完璧」など）は使わず、トレードオフや注意点を正直に書く。

## 4. スニペット (bits) のフロントマター・本文

```markdown
---
type: blurb
date: 2026-07-07T09:00:00+09:00
tags:
  - go
  - context
---

（1〜3文の短いTips。コードブロックを1つ含めてもよい）
```

- `type: blurb` 固定。titleは付けない。
- dateはタイムゾーン付きRFC3339形式（`+09:00`）。

## 5. セルフレビュー（必須）

書き終えたら本文を読み返し、以下を確認する（実際に過去の執筆で起きたミス）。

- 「◯種類」「◯つの方法」などの数と、直後に列挙している項目数が一致しているか。
- Goのバージョン表記が正確か。「Go 1.22で導入された」機能なら「Go 1.21までは〜できなかった」と書く（「Go 1.22までは」は矛盾する）。
- `package main` のコード例は、可能なら一時ディレクトリに抜き出して `go run` でコンパイル・実行できることを確認する。

## 6. 検証（必須）

記事作成後、必ずビルドとフォーマットチェックを行い、警告が出ないことを確認する。

```bash
npm run build
npx prettier --check content/blog/<slug>.md
```

- `icon partial: missing icon` の警告が出たらアイコン名が間違っている。`assets/icons/ui/` の実在ファイルに直す。
- prettierが失敗したら `npx prettier --write` で整形する。
- `ls public/blog/ | grep <slug>` で新記事がビルド出力に**実際に含まれている**ことを確認する。含まれていない場合は日付が未来になっている（`hugo list future` で確認できる）。
- ビルド成果物 `public/` はコミットしない。
